Race Condition Types

Limit-Overrun Race Conditions

Description: Traditional race conditions often involve limit-overrun exploits where synchronized requests are used to overcome specific limits or security checks.

The period of time during which a collision is possible is known as the "race window". This could be the fraction of a second between two interactions with the database, for example.

Like other logic flaws, the impact of a race condition is heavily dependent on the application and the specific functionality in which it occurs.

Example Scenarios:

  • Redeeming a gift card multiple times
  • Repeatedly applying a single discount code
  • Rating a product multiple times
  • Withdrawing or transferring cash in excess of your account balance
  • Reusing a single CAPTCHA solution
  • Bypassing an anti-bruteforce rate-limit

Multi-Endpoint Race Conditions

Involve simultaneous requests to multiple endpoints.

Example:

Cart manipulation in online stores.

Testing Challenges:

  • Vulnerabilities in payment validation and order confirmation within a single request.
  • Alignment Issues in Race Windows:
    • Difficulty syncing race windows for each request.
    • Caused by network delays and varying endpoint processing times.
  • Addressing Network Delays:
    • Use "connection warming" to smooth out delays.
    • Helps distinguish connection setup delays from endpoint-specific delays.
  • Testing with Burp Repeater:
    • Utilize "Send group in sequence" for single connection testing.
    • Longer processing times in the first request may be offset by subsequent requests, indicating successful connection warming.

Single-Endpoint Race Conditions

Involve sending parallel requests with different values to a single endpoint.

Example:

Password reset mechanism storing user ID and reset token in the user's session.

Potential Collision Scenario:

In a password reset mechanism:

  • Two parallel requests from the same session.
  • Different usernames in each request.
  • Collision in the handling of user IDs and reset tokens.

Partial Construction Race Conditions

Occur when applications create objects in multiple steps, introducing a temporary exploitable middle state.

Example:

Registration process creating a user and setting API key in separate steps.

Exploitation Window:

For instance, in user registration:

  • Two separate steps: user creation and API key assignment.
  • Exploitable window where the user exists but API key is uninitialized.

Exploitation Method:

Injecting input to match uninitialized database values.

Exploiting security controls that compare against uninitialized states.
