Preventing Race Condition Vulnerabilities

When a single request can transition an application through invisible sub-states, understanding and predicting its behavior is extremely difficult. This makes defense impractical. To secure an application properly, we recommend eliminating sub-states from all sensitive endpoints by applying the following strategies:

  1. Avoid Mixing Data from Different Storage Places:

    Keep data consistent and avoid mixing information from different storage locations.

  2. Ensure Atomic State Changes:

    Make state changes atomic by leveraging the concurrency features of the datastore. For example, use a single database transaction to check that the payment matches the cart value and confirm the order.

  3. Utilize Datastore Integrity and Consistency Features:

    As a defense-in-depth measure, leverage datastore features such as column uniqueness constraints. These features enhance integrity and consistency in the application's data.

  4. Do Not Rely on One Data Storage Layer to Secure Another:

    Avoid relying on one data storage layer (e.g., sessions) to prevent specific types of attacks (e.g., limit overrun attacks on databases). Each storage layer should be independently secured based on its specific requirements and vulnerabilities.

  5. Ensure Session Handling Framework Consistency:

    Maintain internal consistency in session handling frameworks. Avoid updating session variables individually for the sake of optimization, as this may introduce vulnerabilities. Similarly, with Object-Relational Mapping (ORM) tools, be cautious about hiding critical concepts like transactions, ensuring a clear understanding and responsibility for them.
